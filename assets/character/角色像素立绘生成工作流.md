# 角色像素初始立绘生成工作流

本文档用于后续批量生成 IP 角色初始站立立绘。用户只需要说明“IP + 角色”，执行者应按本文档直接联网核查官方形态、编写提示词、单形态绘制、后处理、审核并落地到 `main/assets/character`。如果用户没有指定形态范围，默认由执行者联网检索并筛选官方主线形态；如果用户指定了形态范围，则在官方资料内收窄执行。

## 目标

- 为游戏生成可直接作为 Godot 运行时资源使用的角色初始站立立绘。
- 每个角色的每个形态单独成图、单独建文件夹。
- 只绘制官方主线或官方资料中存在的形态，不采用同人漫画、原创皮肤、活动换装、非主线动画原创形态，除非用户明确要求。
- 所有 IP 的输出都要尽量统一到同一种站姿、比例和像素立绘风格。

## 根目录

运行时角色资产根目录：

```text
D:\游戏\earth-online\main\assets\character
```

现有 IP 分类示例：

```text
assets/character/
  七龙珠/
  火影忍者/
```

后续新增 IP 时，在 `character` 下新建同名文件夹。例如：

```text
assets/character/海贼王/
assets/character/死神/
assets/character/咒术回战/
```

## 目录命名规范

普通角色：

```text
assets/character/<IP>/<角色slug>/<形态名>/<角色slug>.png
```

需要进一步分组时：

```text
assets/character/<IP>/<分组>/<角色slug>/<形态名>/<角色slug>.png
```

示例：

```text
assets/character/火影忍者/naruto/形态01_常态/naruto.png
assets/character/火影忍者/历代影/木叶隐村/hashirama_senju/形态02_秽土转生/hashirama_senju.png
```

命名规则：

- IP 和形态文件夹可用中文，角色 slug 用小写英文、下划线。
- 形态文件夹统一使用 `形态NN_名称`，例如 `形态01_常态`、`形态02_秽土转生`。
- 同一角色的最终 PNG 文件名固定为 `<角色slug>.png`。
- 每个角色或分组可生成 `*_forms_preview.png` 预览图。

## 风格基准

首要参考图：

```text
D:\游戏\earth-online\main\assets\character\七龙珠\wukong\形态A\wukong.png
```

统一风格要求：

- 单角色、全身、初始站立立绘。
- 正面或轻微 3/4 正面站姿，脚底接近统一底部锚点。
- 目标画布 `256x256`，RGBA 透明背景。
- 人形角色主体高度通常约 `220-225px`，矮小角色可保留设定差异，但要保证可读性。
- 深色外轮廓，粗颗粒像素块面，低抗锯齿，避免过于平滑的高清动漫插画感。
- 角色身份特征优先清楚：发型、服装、武器、眼睛、标志性轮廓。

## 官方形态筛选

每次开始绘制前必须联网核查。

推荐核查顺序：

1. 官方网站、官方图鉴、官方游戏资料、官方设定集页面。
2. 可靠资料站作为交叉核对，例如 Narutopedia、Dragon Ball Wiki 等。
3. 图片搜索只用于辅助观察外观，不作为形态官方性的唯一依据。

筛选原则：

- 只纳入官方主线、官方动画、官方漫画、官方游戏或官方设定中明确出现的形态。
- 同人漫画、二创融合、非官方配色、玩家 MOD、活动皮肤默认排除。
- “短暂出现但官方可见”的形态可以绘制。
- “只有名字、没有官方外观”的形态不要补同人设定。
- 秽土转生、机械化、黄金化、黑化、六道化等特殊形态必须确认官方出现过。
- 如果存在“动画原创但非漫画主线”的争议形态，除非用户明确要求，否则先排除并在提示词文件写明。

## 提示词文件

每个 IP 或大型批次都要写一个提示词 Markdown，放在对应 IP 目录中。

示例：

```text
assets/character/火影忍者/official_prompts_villains.md
assets/character/火影忍者/official_prompts_kage.md
```

提示词文件应包含：

- 本批次范围说明。
- 纳入形态清单。
- 排除形态清单。
- 通用风格模板。
- 每个角色/形态的独立英文绘图提示词。
- 官方性备注，尤其是容易误判的形态。
- 证据表：每个角色/形态对应的来源 URL、来源类型、访问日期、纳入或排除理由。

证据表建议格式：

```markdown
| 角色 | 形态 | 结论 | 来源 URL | 来源类型 | 访问日期 | 备注 |
|---|---|---|---|---|---|---|
| 示例角色 | 形态01_常态 | 纳入 | https://... | 官方站/资料站 | 2026-07-07 | 官方主线可见 |
| 示例角色 | 动画原创形态 | 排除 | https://... | 资料站 | 2026-07-07 | 非主线动画原创 |
```

## 绘图提示词模板

默认使用内置 `image_gen` 逐张生成。不要把多个角色或多个形态放在同一张原图中。

通用模板：

```text
Create ONE full-body 2D pixel-art game sprite of <character and official form>, official <IP> form, using the Wukong sprite shown earlier as the visual style and stance reference: compact 256x256-ready pixel-art standing portrait, front 3/4 facing viewer, feet near bottom anchor, crisp dark outline, hard-edged chunky pixel clusters, low anti-aliasing, readable anime costume details, not smooth high-resolution illustration.

Subject details: <official identity markers: hair, eyes, costume, weapon, aura, body state>.

Composition: centered full body, front torso visible, both shoulders visible, contained inside the frame with generous padding. Keep weapons, hair, capes, orbs, tails, wings, and effects close to the body. No extra characters, no text anywhere.

Background: perfectly flat solid #00ff00 chroma-key background, one uniform color, no shadows, no gradients, no texture, no floor plane. Do not use #00ff00 anywhere in the character. No watermark, no signature.
```

常见负面约束：

- `No text, no kanji, no letters, no numbers, no symbols, no printed marks, no watermark.`
- `Do not draw a back view, rear cloak, side profile, or turned-away head.`
- `Do not include extra characters, summons, large aura, ground, scene background, or cast shadow.`
- `If a mark may look like a number or letter, simplify or omit it.`

## 生成规则

- 每个形态必须单独调用一次图像生成。
- 不要一次性生成一整排角色，也不要生成角色合集再裁切。
- 如果生成结果出现背身、错角色、文字、明显同人化、形态混合，立即重抽该单张。
- 对披风、护额、葫芦、卷轴、胸甲等容易出现文字的位置，要明确写 `plain, no text-like marks`。
- 对绿色角色，仍优先使用 `#00ff00` 色键；如果角色主体大面积接近绿色，可临时改用其他纯色键，但必须做残留检查。

## 后处理标准

生成原图后，本地执行色键抠图和标准化：

- 输入：纯色 `#00ff00` 背景原图。
- 输出：`256x256 RGBA` 透明 PNG。
- 边缘：透明边 alpha 必须为 `0`。
- 锚点：人形角色 bbox 底部通常统一到 `y=240`。
- 绿色/洋红色键残留：最终可见像素中应为 `0`。
- 预览图可使用深色背景拼接，但单张角色 PNG 必须透明。

后处理应做：

- 只移除与边界连通的色键背景，避免误删角色内部颜色。
- 进行轻度 despill，清掉绿色边缘污染。
- 按统一高度缩放，保持角色完整。
- 不用脚本生成创意美术，只能用脚本做抠图、缩放、锚点、拼预览和 QC。

建议保留的中间产物和元数据：

```text
assets/character/<IP>/<角色slug>/<形态名>/
  <角色slug>.png              # 最终透明 PNG，游戏实际使用
  raw-source.png              # 可选：本形态选中的原始生成图副本
  prompt-used.txt             # 可选：本形态实际使用的最终提示词
  qc-meta.json                # 可选：尺寸、bbox、边缘 alpha、色键残留等检查结果
```

如果批次很大，也可以将原图和 QC 汇总放在分组目录：

```text
assets/character/<IP>/_generation_logs/<批次名>/
  source_map.csv
  qc_summary.json
  rejected_sources/
```

不要把废稿当作正式形态文件；废稿只能放在 `_generation_logs` 或直接保留在 Codex 默认生成目录。

## 子智能体分工

用户要求批量绘制时，应采用子智能体并行处理和统一检查。

推荐分工：

- 主线程：确定范围、写提示词文件、维护总清单、最终整合、最终检查。
- 子智能体 A：官方形态审核，核对纳入/排除清单，确认是否存在非官方或漏项。
- 子智能体 B：视觉风格审核，对比悟空形态A，检查站姿、比例、文字、水印、色键残留。
- 子智能体 C：可按角色组或阵营分包绘制，但必须拥有互不重叠的输出目录，避免写入冲突。

并行绘制规则：

- 按 IP 下的自然分组分包，例如村、阵营、角色线、敌方组织。
- 每个子智能体只能写自己负责的角色目录。
- 预览汇总图和总 QC 由主线程统一生成。
- 子智能体不得重写其他分包的文件。
- 主线程必须复核子智能体输出后再交付。

## 统一检查清单

每批完成后必须检查：

- 形态数量是否等于清单数量。
- 每个形态是否有独立文件夹。
- 每张单图是否为 `256x256 RGBA`。
- bbox 底部是否统一，通常为 `240`。
- 画布边缘 alpha 是否全为 `0`。
- 可见像素是否有绿色/洋红色键残留。
- 是否存在文字、数字、水印、签名。
- 是否为正面或 3/4 正面站姿。
- 是否存在错角色、错形态、同人配色、非官方形态。
- 预览图是否能一眼看出同批次风格一致。

机器检查建议输出：

```text
checked_forms=<数量>
errors=0
edge=0
green_key=0
magenta_key=0
```

如果发现问题：

- 硬错误必须返工：错角色、错形态、文字、水印、背身、抠图残留、画布裁切。
- 软问题可记录：像素感略弱、角色设定导致身高差、早期官方造型资料少。

## 交付内容

最终回复应包含：

- 提示词文件路径。
- 资产根目录路径。
- 本批次角色/形态数量。
- 是否联网核查官方形态。
- 是否使用子智能体审核。
- 技术检查结果。
- 预览图 Markdown 图片链接。
- 明确说明未纳入的争议形态或排除项。
